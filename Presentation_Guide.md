### הבעיה

### הפתרון
- מערכת לבישה עם חיישן תאוצה + עיבוד אותות דיגיטלי
- אב-טיפוס בקוד פתוח - מאפשר שיפור אלגוריתם עתידי
- אימות המערכת: מנוע DC עם מסה לא מאוזנת כסימולטור מכני (כי לא ניתן לנסות על חולים)

## שקף 5 - שיטת הכיול (Calibration)

### הבעיה
- חיישן ה-MPU6050 אינו מושלם: יש offset (היסט) בכל ציר
- בלי כיול, ציר Z (כלפי מעלה) יקרא ~10.46 m/s² במקום 9.81 (כבידה)
- offset זה ייצור רכיב DC שיסתיר את הרעד

### שיטת הכיול שבוצעה: כיול סטטי (Static Calibration)
1. **הנחת החיישן על משטח ישר ויציב** (ציר Z מצביע למעלה)
2. **דגימת ~1000 ערכים** במנוחה מוחלטת
3. **חישוב ממוצע** לכל ציר:
   - ציר X: צפי 0, נמדד ~0.301 → offset = **0.301 m/s²**
   - ציר Y: צפי 0, נמדד ~0.016 → offset = **0.016 m/s²**
   - ציר Z: צפי 9.81, נמדד ~10.46 → offset = 10.46-9.81 = **1.046 m/s²** (ביחס לכבידה)
4. **שמירת הoffsets בקוד** (קבועים בקושחה):
   ```c
   float aX_off = 0.301009, aY_off = 0.016101, aZ_off = 1.046231;
   ```
5. **בזמן דגימה**: כל ערך מתוקן: `ax = a.acceleration.x - aX_off`

### למה כיול סטטי מספיק?
- הרעד שאנחנו מודדים הוא **תנודתי** (AC) - לא DC
- גם אם נשאר offset קטן, מסנן ה-Bandpass (2-8 Hz) בעיבוד מסיר אותו (DC = 0 Hz מונחת לחלוטין)
- כיול דינמי (6-point calibration) הוא מורכב יותר ולא נדרש ליישום הזה

## שקף 6 - ערוץ התקשורת: UART over USB

### ⚠️ יישוב סתירה מהספר
> בהצעת הפרויקט תוכנן ערוץ WiFi. בפועל הוחלף ל-**UART over USB**.
> זו **החלטה הנדסית מודעת** שנעשתה לאחר התנסות וייעוץ עם המנחה.

### למה לא WiFi?
- אי-יציבות רשת WiFi במכללה גרמה לאיבוד חבילות
- ב-100Hz דגימה, כל חבילה שאבדה = "חור" בנתונים
- עיבוד אותות (FFT/PSD) דורש **רצף נתונים ללא פערים**

### למה UART over USB?
| פרמטר | ערך |
|--------|------|
| Baud Rate | 115,200 bps |
| פורמט שורה | `timestamp,Ax,Ay,Az\n` (~30 bytes) |
| קצב נדרש | 100 שורות/שנייה × 30 bytes = **3,000 B/s** |
| קצב זמין | 115,200 / 10 bits = **11,520 B/s** |
| **ניצולת** | **~26%** - מרווח ביטחון רחב |

### יתרונות
- **אמינות 100%**: חיבור קווי, אין איבוד חבילות
- **פשטות**: אין צורך בפרוטוקול TCP/IP, buffering, reconnect
- **צריכת חשמל**: אין WiFi radio = פחות חשמל
- **חסרון**: כבל פיזי בין ESP32 ל-RPi (מתקבל במערכת מעבדה)

### הערה לגבי FIFO
> הספר מציין "אחסון זמני ב-FIFO buffer". בפועל, הקוד דוגם ושולח ישירות ב-Serial
> בכל sample (כל 10ms). ה-UART hardware buffer של ESP32 (128 bytes) משמש
> כ-FIFO טבעי ברמת החומרה - מספיק בקצב 3KB/s.

## שקף 8 - מערכת האימות המכני: מנוע + L298N

> **קובץ קוד**: `motor_control.py`

### L298N Dual H-Bridge Motor Driver
- **תפקיד**: מגשר בין הלוגיקה 3.3V של RPi למנוע 12V DC
- **מתח מקסימלי**: 35V
- **זרם מקסימלי**: 2A רציף, 3A רגע
- **שליטה**: PWM ב-Enable (מהירות) + IN1/IN2 (כיוון)
- **[Datasheet](https://www.handsontec.com/dataspecs/L298N%20Motor%20Driver.pdf)**

### חיבורי GPIO
| פין RPi | חיבור ל-L298N | תפקיד |
|---------|---------------|--------|
| GPIO 18 | ENA (Enable A) | PWM - שליטה במהירות |
| GPIO 23 | IN1 | כיוון סיבוב (HIGH = קדימה) |
| GPIO 24 | IN2 | כיוון סיבוב (LOW = קדימה) |

### מנוע DC - JGA 25-370
- **מתח**: 12V DC
- **מהירות מקסימלית**: 625 RPM (ללא עומס)
- **זרם**: 0.04-2A
- **מסה לא מאוזנת**: 40g ברדיוס 1.5 cm מציר הסיבוב

### טבלת המרה: Duty Cycle → מהירות → תדר רעד
| Duty Cycle | מתח ממוצע | RPM | Hz (תדר רעד) |
|-----------|-----------|-----|--------------|
| 20% | 2.4V | 125 | **2.1 Hz** |
| 30% | 3.6V | 188 | **3.1 Hz** ← תחילת טווח פרקינסון |
| 40% | 4.8V | 250 | **4.2 Hz** |
| 50% | 6.0V | 312 | **5.2 Hz** ← מרכז הטווח |
| 60% | 7.2V | 375 | **6.3 Hz** |
| 70% | 8.4V | 438 | **7.3 Hz** ← סוף טווח פרקינסון |
| 80% | 9.6V | 500 | **8.3 Hz** |

### שיטת בקרת PWM
- **Software PWM** דרך RPi.GPIO library (לא hardware PWM)
- **תדר נושא**: 1 kHz (הרבה מעל תדר הרעד, מונע "גמגום")
- **בקרה**: open-loop (לולאה פתוחה) - אין חיישן encoder על המנוע
- ההמרה duty→RPM היא **לינארית** (0%=0 RPM, 100%=625 RPM)

## שקף 10 - שיטת המדידה: למה 100 Hz? למה 2 דקות?

### למה קצב דגימה של 100 Hz?

**תשובה מבוססת Nyquist:**
- רעד מנוחה: 3-7 Hz
- משפט Nyquist: דגימה מינימלית = 2 × fmax = 2 × 7 = **14 Hz**
- אבל 14 Hz זה המינימום המתמטי בלבד. בפועל:

| סיבה | הסבר |
|-------|------|
| מסנן אנטי-aliasing | המסנן שלנו מגיע ל-8Hz. צריך מרווח מעל 8Hz כדי שהמסנן יספיק לנחית |
| רזולוציית זמן | ב-100Hz כל sample = 10ms. ב-14Hz כל sample = 71ms - מאבדים פרטים |
| עמידות לרעש | יותר samples = יותר ממוצע = פחות רעש |
| תקן מקובל | מחקרים על רעד פרקינסון (MDPI refs) משתמשים ב-50-200Hz |

**שאלה: האם 50 Hz היה מספיק?**
- מתמטית: כן. 50Hz > 2×8Hz (ה-Nyquist של הסרט המורחב)
- מעשית: 100Hz נותן ביטחון נוסף, ועומס ה-bandwidth (3KB/s) לא מהווה בעיה
- **מסקנה**: 50Hz היה עובד, 100Hz עדיף כי אין מחיר אמיתי

### למה 2 דקות של איסוף?

**תשובה מבוססת Welch PSD:**
- חלון Welch = 4 שניות, חפיפה 50%
- ב-120 שניות: (120-4)/(4×0.5) + 1 = **~58 חלונות**
- ממוצע על 58 חלונות → PSD חלק ואמין

**שאלה: האם 30 שניות היו מספיקות?**
- ב-30 שניות: (30-4)/2 + 1 = **~14 חלונות** - עדיין סביר
- אבל 58 חלונות >>> 14 חלונות מבחינת הפחתת רעש
- בנוסף: 2 דקות מאפשרות לראות **שינויים לאורך זמן** (מודולציית אמפליטודה)
- **מסקנה**: 30 שניות יעבוד, 120 שניות נותן תוצאה אמינה הרבה יותר

### רזולוציית תדר ב-PSD
```
δf = 1/T_window = 1/4s = 0.25 Hz
```
- זו הסיבה שב-PSD התדר המזוהה (למשל 5.75Hz) לא בדיוק שווה לצפוי (5.2Hz)
- ההפרש 0.55Hz הוא בגדר שגיאת הדגימה (2 bins × 0.25Hz)

## שקף 11 - אלגוריתם עיבוד האותות (DSP Pipeline)

```
   נתונים גולמיים (Ax, Ay, Az כולל כבידה)
              │
              ▼
   ┌─────────────────────┐
   │ 1. Bandpass Filter  │  Butterworth Order 4, 2-8 Hz
   │    (filtfilt)       │  מופעל על כל ציר בנפרד
   │                     │  מסיר DC (כבידה) + רעש גבוה
   └──────────┬──────────┘
              ▼
   ┌─────────────────────┐
   │ 2. Resultant Vector │  R = √(Ax_f²+Ay_f²+Az_f²)
   │    (מצירים מסוננים) │  → RMS = עוצמת רעד כוללת
   └──────────┬──────────┘
              ▼
   ┌─────────────────────┐
   │ 3. PSD (Welch)      │  על כל ציר מסונן בנפרד
   │    + בחירת ציר      │  → ציר דומיננטי (פיק PSD מקסימלי)
   │      דומיננטי       │  → תדר דומיננטי, DPR, הספק בפס
   └──────────┬──────────┘
              ▼
   ┌─────────────────────┐
   │ 4. FFT              │  ספקטרום גולמי (0-12Hz) על כל 120 שניות
   │    (על אות RAW)     │  → אישור התדר + זיהוי הרמוניות + תוכן מחוץ לפס
   └──────────┬──────────┘
              ▼
   ┌─────────────────────┐
   │ 5. Spectrogram      │  STFT על אות גולמי (0-12Hz)
   │    (על אות RAW)     │  → יציבות התדר לאורך זמן
   └─────────────────────┘
```

### למה Bandpass ולא DC Removal (הסרת ממוצע)?
- הסרת ממוצע מנטרלת רק DC **קבוע** — לא מטפלת בסיבוב כף היד
- מסנן Bandpass 2-8 Hz מסיר **כל** מה שמתחת ל-2 Hz (כולל כבידה, תנועות איטיות)
- בנוסף מסנן רעש גבוה (מעל 8 Hz) — הכל בצעד אחד
- הסינון **לפני** חישוב הוקטור השקול מונע עיוות לא-ליניארי

### למה filtfilt (Zero-Phase)?
- מסנן רגיל (causal) מזיז את האות בזמן (phase shift)
- filtfilt מעביר קדימה **ואחורה** → מבטל phase shift
- **חשוב**: עובד רק offline (לא real-time) → מתאים למערכת שלנו
- האפקט: סדר אפקטיבי = 8 (כפול מ-4), אטנואציה חדה יותר

### למה מסנן 2-8 Hz ולא 3-7 Hz?
- בתדר הקיטעון (cutoff) של מסנן Butterworth יש הנחתה של -3dB
- אם המסנן היה 3-7Hz, רעד בדיוק ב-3Hz או 7Hz היה מונחת ב-30%!
- **הפתרון**: הרחבה ל-2-8Hz → טווח 3-7Hz נמצא **באזור השטוח** של המסנן
- תדרים מתחת ל-2Hz (כבידה, תנועות איטיות) עדיין מסוננים

## שקף 13 - מכניקת הרעד: כוח צנטריפוגלי

### עקרון
מסה m=40g מסתובבת ברדיוס r=1.5cm מציר המנוע.
הכוח הצנטריפוגלי:
```
F = m · ω² · r = m · (2πf)² · r = 4π² · m · f² · r
F = 0.0236 · f²  [N]
```

### תאוצה על כף היד
```
a(f) = F / M_total = 0.0236·f² / 0.65 = 0.04·f²  [m/s²]
```
(M_total = 0.65 kg = מנוע + כף יד)

### בדיקת SNR: האם האות חזק מספיק?
| תדר | תאוצה a(f) | רצפת רעש חיישן | יחס |
|------|-----------|----------------|------|
| 3 Hz | 0.36 m/s² | ±0.1 m/s² | **×3.6** ✓ |
| 5 Hz | 1.0 m/s² | ±0.1 m/s² | **×10** ✓ |
| 7 Hz | 1.96 m/s² | ±0.1 m/s² | **×20** ✓ |

**מסקנה**: גם בתדר הנמוך ביותר (3Hz), האות חזק פי 3.6 מרצפת הרעש.

## שקף 18 (גיבוי) - שאלות צפויות ותשובות

**ש: למה לא השתמשתם בגירוסקופ של ה-MPU6050?**
ת: רעד מנוחה מתבטא בתנודות **תאוצה ליניארית** (לא סיבוב). חיישן תאוצה מספיק ונותן מדידה ישירה של הכוח. גירוסקופ היה רלוונטי אם היינו רוצים לזהות סוג התנועה (rotation vs translation).

**ש: למה לא real-time processing?**
ת: filtfilt דורש את **כל האות** כדי לעבד קדימה ואחורה. בreal-time אפשר רק מסנן causal שגורם ל-phase shift. לאנליזה קלינית, offline מדויק יותר.

**ש: מה יקרה עם חולה אמיתי?**
ת: הרעד של חולה פרקינסון דומה בתדר (3-7Hz) אבל באמפליטודה קטנה יותר. ה-SNR יהיה נמוך יותר, אבל החיישן רגיש מספיק (הראינו זיהוי ב-3Hz עם יחס ×3.6).

**ש: למה open-loop על המנוע?**
ת: בקרה open-loop מספיקה כי אנחנו לא צריכים תדר מדויק לHz - אנחנו צריכים **ידוע בערך** כדי לאמת שהMeasured קרוב ל-Expected. ההפרש בין 5.2 ל-5.75 Hz מוסבר ברזולוציית PSD (0.25Hz bins).

### A.4 — מנתח (Analyzer): offline_analyzer_exp.py — נוסחאות DSP

#### שלב 1: טעינת נתונים
```
קלט: CSV עם עמודות [Timestamp, Ax, Ay, Az]
t = Timestamp / 1000   (המרה ממילישניות לשניות)
```

#### שלב 2: סינון פס-רחב (Bandpass Filter)
**מסנן Butterworth סדר 4, פס 2-8 Hz, מוחל עם filtfilt:**

תדר Nyquist:
```
f_nyquist = Fs / 2 = 100 / 2 = 50 Hz
```

תדרי קיטעון מנורמלים:
```
Wn_low  = f_low  / f_nyquist = 2 / 50 = 0.04
Wn_high = f_high / f_nyquist = 8 / 50 = 0.16
```

פונקציית התמסורת של מסנן Butterworth סדר N:
```
|H(f)|² = 1 / (1 + (f/fc)^(2N))
```

**filtfilt**: מעביר את האות פעמיים — קדימה ואחורה:
```
H_effective(f) = H(f) · H*(f) = |H(f)|²
```
- סדר אפקטיבי: 2×4 = **8**
- הנחתה אפקטיבית בתדר הקיטעון: **-6 dB** (במקום -3 dB)
- **Phase = 0** לכל תדר (zero-phase filter)

הסינון מוחל על **כל ציר בנפרד**:
```
Ax_filt = filtfilt(b, a, Ax)
Ay_filt = filtfilt(b, a, Ay)
Az_filt = filtfilt(b, a, Az)
```

**למה Bandpass מסיר DC (כבידה)?**
- DC = תדר 0 Hz → מונחת לחלוטין ע"י המסנן (הנחתה > 80 dB ב-0 Hz)
- אין צורך בהסרת ממוצע נפרדת — ה-Bandpass עושה הכל בשלב אחד
- גם תנודות איטיות (drift, תנועת יד < 2 Hz) מסוננות

#### שלב 3: וקטור שקול (Resultant Vector) — מהצירים המסוננים
```
R_filtered[i] = √(Ax_filt[i]² + Ay_filt[i]² + Az_filt[i]²)
```
זהו הגודל (magnitude) של וקטור התאוצה לאחר סינון — מייצג את **עוצמת הרעד הכוללת** בכל רגע.

**וקטור שקול גולמי** (לתצוגה בלבד, לא לחישוב מדדים):
```
R_raw[i] = √(Ax[i]² + Ay[i]² + Az[i]²)
R_raw_dc_removed = R_raw - mean(R_raw)
```
הסרת ממוצע מהוקטור הגולמי נעשית רק כדי להציג את הגרף ממורכז סביב 0.

#### שלב 4: RMS — עוצמת רעד כוללת

**RMS מחושב על הוקטור השקול של הצירים המסוננים:**
```
RMS = √(1/N · Σᵢ R_filtered[i]²)
    = √(1/N · Σᵢ (Ax_filt[i]² + Ay_filt[i]² + Az_filt[i]²))
    = √(RMS_x² + RMS_y² + RMS_z²)
```

כאשר:
```
RMS_x = √(1/N · Σ Ax_filt[i]²)
RMS_y = √(1/N · Σ Ay_filt[i]²)
RMS_z = √(1/N · Σ Az_filt[i]²)
```

**פירוש**: ה-RMS הוא **השורש הריבועי של הממוצע של ריבועי הוקטור השקול**.
- מייצג את עוצמת הרעד הכוללת ב-3 צירים
- יחידות: m/s²
- מבוסס על **צירים מסוננים** (2-8 Hz בלבד) → מודד רק אנרגיית רעד, לא כבידה/רעש

#### שלב 5: PSD — צפיפות הספק ספקטרלית (Welch)

PSD מחושב על **כל ציר מסונן בנפרד** (לא על הוקטור השקול):
```
[f, PSD_x] = Welch(Ax_filt, Fs=100, nperseg=400, noverlap=200)
[f, PSD_y] = Welch(Ay_filt, Fs=100, nperseg=400, noverlap=200)
[f, PSD_z] = Welch(Az_filt, Fs=100, nperseg=400, noverlap=200)
```

פרמטרי Welch:
```
חלון: 4 שניות = 400 דגימות (nperseg)
חפיפה: 50% = 200 דגימות (noverlap)
רזולוציית תדר: δf = 1/T_window = 1/4 = 0.25 Hz
מספר חלונות ב-120s: (120-4)/(4×0.5) + 1 ≈ 58 חלונות
```

#### שלב 6: בחירת ציר דומיננטי

הציר הדומיננטי הוא זה שיש לו את **הפיק הגבוה ביותר ב-PSD** בטווח 2-8 Hz:
```
dominant_axis = argmax over {X,Y,Z} of max(PSD_axis[2-8 Hz])
```

כל מדדי התדר (תדר דומיננטי, DPR, הספק בפס) מחושבים מה-PSD של **הציר הדומיננטי בלבד**.

#### שלב 7: תדר דומיננטי (Dominant Frequency)
```
f_dominant = f[argmax(PSD_dominant[2-8 Hz])]
```
התדר בו הפיק הגבוה ביותר ב-PSD — זהו **תדר הרעד** שהמערכת מדווחת.

#### שלב 8: הספק כולל בפס (Total Band Power)
```
P_total = ∫₂⁸ PSD_dominant(f) df   (אינטגרציה מספרית עם trapz)
```
מייצג את סך האנרגיה בטווח 2-8 Hz של הציר הדומיננטי.

#### שלב 9: DPR — יחס הספק דומיננטי (Dominant Power Ratio)
```
P_peak = ∫(f_peak-δf)^(f_peak+δf) PSD_dominant(f) df
DPR = P_peak / P_total
```
כאשר δf = 0.25 Hz (רוחב bin אחד).

**פירוש**: כמה מהאנרגיה מרוכזת סביב הפיק. DPR גבוה (>50%) = רעד "נקי" בתדר אחד.

#### שלב 10: סטיית תדר (Frequency Deviation)
```
deviation = |f_dominant - f_pwm|   [Hz]
```
כאשר f_pwm = תדר המנוע הצפוי (קלט מהמשתמש). מדד אינפורמטיבי בלבד — **אין PASS/FAIL**.

#### שלב 11: FFT — ספקטרום על 120 שניות מלאות

FFT מחושב על **אות הציר הדומיננטי המסונן**:
```
X[k] = Σₙ x[n] · e^(-j·2π·k·n/N)
|X[k]| = magnitude (amplitude spectrum)
f[k] = k · Fs / N
```
כאשר N = מספר הדגימות הכולל (~12,000), Fs = 100 Hz.
רזולוציית תדר: δf = Fs/N = 100/12000 ≈ 0.0083 Hz (הרבה יותר עדינה מ-PSD).

#### שלב 12: Hilbert Envelope (מעטפת)
```
envelope[i] = |Hilbert(dominant_filt)|
```
מוצגת כקווים מקווקווים בגרף Fig 2.2. מראה את **מודולציית האמפליטודה** של הרעד לאורך זמן.

### A.5 — סיכום: מה מחושב מאיפה?

| מדד | מקור הנתונים | נוסחה | יחידות |
|------|--------------|--------|--------|
| **RMS** | וקטור שקול מסונן (3 צירים) | √(mean(R²)) | m/s² |
| **Dominant Freq** | PSD של ציר דומיננטי | f[argmax(PSD)] | Hz |
| **Total Power** | PSD של ציר דומיננטי | ∫PSD df | (m/s²)²/Hz |
| **Peak Power Density** | PSD של ציר דומיננטי | max(PSD) | (m/s²)²/Hz |
| **DPR** | PSD של ציר דומיננטי | P_peak / P_total | % |
| **Deviation** | תדר דומיננטי vs PWM | \|f_dom - f_pwm\| | Hz |
| **FFT** | אות ציר דומיננטי מסונן | DFT magnitude | m/s² |
| **Envelope** | אות ציר דומיננטי מסונן | \|Hilbert(x)\| | m/s² |

### A.6 — מנוע DC: נוסחאות אימות

#### כוח צנטריפוגלי
```
F = m · ω² · r = m · (2πf)² · r = 4π² · m · f² · r
```
עם m=0.04 kg, r=0.015 m:
```
F = 4π² × 0.04 × 0.015 × f² = 0.0237 · f²  [N]
```

#### תאוצה צפויה
```
a(f) = F / M_total = 0.0237·f² / 0.65 ≈ 0.036·f²  [m/s²]
```
(M_total ≈ 0.65 kg = כף יד + מנוע + חיישן)

#### SNR מינימלי
```
SNR = a(f) / noise_floor
noise_floor ≈ 0.1 m/s² (MPU6050 @ ±2g, 100 Hz BW)
```

### A.7 — סינון Bandpass כהסרת DC: הסבר מתמטי

**למה Bandpass Filter מספיק ולא צריך הסרת ממוצע נפרדת?**

אות מד-תאוצה גולמי:
```
x(t) = g·cos(θ) + offset + tremor(t) + noise(t)
        ────────   ──────   ─────────   ────────
        DC (כבידה)  כיול     2-8 Hz      רעש
        ~9.81 m/s²  ~0.3     ~0.5 m/s²   >8 Hz
```

לאחר Bandpass 2-8 Hz:
```
x_filtered(t) ≈ tremor(t)
```
- DC (0 Hz): הנחתה > 80 dB → מוכפל ב-0.0001
- Drift (< 2 Hz): הנחתה > 40 dB
- Noise (> 8 Hz): הנחתה > 40 dB
- **tremor (2-8 Hz): מועבר ב-0 dB (ללא שינוי)**

הערה: בקוד, `result_raw - mean(result_raw)` (שורה 369) נעשה רק לצורך **הצגה גרפית** של הוקטור הגולמי ממורכז סביב 0. זו **לא** חלק מהעיבוד — כל החישובים המשמעותיים (RMS, PSD, FFT) מבוססים על הצירים המסוננים.
